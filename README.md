# graphrank — LeaderRank для графов, не помещающихся в память

Решение тестового задания «Аналитика над графом»: вычисление аналитической
метрики для невзвешенного ориентированного графа при условии, что граф
не помещается в оперативную память одного вычислительного узла, а в графе
могут встречаться гипер-узлы (вершины со степенью на порядки выше средней).

Реализация: C++17, только стандартная библиотека, без внешних зависимостей.
Алгоритм, работа с памятью и многопоточность реализованы в коде проекта.

## Быстрый старт (Ubuntu 22.04, x64)

```bash
sudo apt-get install -y g++ cmake make python3

git clone <этот репозиторий> && cd graph-leaderrank

# сборка (вариант 1: CMake)
cmake -B build -DCMAKE_BUILD_TYPE=Release && cmake --build build
# сборка (вариант 2: без CMake)
make

# запуск на примере из условия задачи
./build/graphrank --input tests/sample/edges.csv --output leader_rank.csv \
                  --memory-mb 128
cat leader_rank.csv

# сквозные тесты (сверка с независимой эталонной реализацией и др.)
./tests/run_tests.sh build/graphrank
```

Запуск на реальном датасете SNAP (парсер понимает формат SNAP напрямую:
табуляции и строки-комментарии `#`):

```bash
wget https://snap.stanford.edu/data/soc-LiveJournal1.txt.gz
gunzip soc-LiveJournal1.txt.gz
./build/graphrank --input soc-LiveJournal1.txt --output lj_rank.csv \
                  --memory-mb 128 --workdir /path/to/ssd/tmp
```

Аргументы командной строки:

| аргумент | значение |
|---|---|
| `--input PATH` | входной CSV/TSV с рёбрами (первые две колонки: from, to) |
| `--output PATH` | выходной CSV `vertex,rank` |
| `--memory-mb N` | бюджет памяти в МБ (минимум 64; по умолчанию 1024) |
| `--workdir DIR` | каталог для временных файлов, желательно на SSD |
| `--threads N` | число потоков (по умолчанию все доступные CPU) |
| `--tol X` | порог сходимости, L1-норма изменения (по умолчанию 1e-9) |
| `--max-iters N` | максимум итераций (по умолчанию 100) |
| `--slice-size N` | принудительный размер слайса вершин — тестовая ручка |
| `--keep-workdir` | не удалять временные файлы после завершения |

Программа печатает в stderr журнал фаз, а на каждой итерации — L1-дельту,
суммарную массу (инвариант корректности, см. ниже) и время. В конце —
топ-10 вершин и **фактический пиковый RSS процесса**, чтобы соблюдение
бюджета памяти можно было проверить, а не принимать на веру.

## 1. Выбор метрики: LeaderRank

**LeaderRank** (Lü, Zhang, Yeung, Zhou, *Leaders in Social Networks, the
Delicious Case*, PLoS ONE, 2011) — метрика важности вершин ориентированного
графа, близкая родственница PageRank. К графу добавляется *ground-вершина*
g, соединённая двунаправленными рёбрами со всеми N вершинами, и по
расширенному графу запускается обычное случайное блуждание:

```
s_{t+1}(v) = Σ_{u→v} s_t(u) / outdeg'(u),   outdeg'(u) = outdeg(u) + 1
```

После сходимости очки ground-вершины распределяются поровну между всеми:
`final(v) = s(v) + s(g)/N`. Метрика применяется ко всему графу и отвечает
на тот же вопрос, что и PageRank: «насколько вершина важна с учётом
важности ссылающихся на неё вершин» (поиск лидеров мнений, ранжирование
страниц/пользователей, выявление влиятельных узлов).

Почему именно она:

1. **Нет свободного параметра.** У PageRank есть damping factor (обычно
   0.85, но это соглашение); LeaderRank параметров не имеет — ground-вершина
   играет роль «телепорта» естественным образом. Результатам проще доверять:
   нечего подбирать.
2. **Проблема висячих вершин решается сама.** Вершина без исходящих рёбер
   в PageRank «теряет» массу, и во внешнепамятной реализации перераспределение
   массы висячих вершин — отдельная головная боль. В LeaderRank каждая
   вершина имеет ребро в ground, висячих вершин не существует по построению.
3. **Устойчивость.** В оригинальной работе показано, что LeaderRank
   устойчивее PageRank к шуму и манипуляциям (спам-ссылкам).
4. **Вычислительно это power iteration по рёбрам** — ровно тот класс
   алгоритмов, для которого имеет смысл строить внешнепамятный движок:
   один и тот же код после косметических правок считает PageRank, Katz и
   другие метрики вида «распространение значения по рёбрам».

Важная деталь реализации: ground-вершина **не материализуется** (это было бы
+2N рёбер на диске). Её вклад учитывается аналитически: каждая вершина
получает слагаемое `s(g)/N`, а приток в g равен сумме всех вкладов
`Σ s(v)/outdeg'(v)` — оба значения считаются попутно за O(V) без единого
дополнительного ребра.

## 2. Варианты представления и обработки графа

Ниже три рассмотренных варианта. Общая для всех идея: итерации вида
`next[dst] += rank[src]/outdeg[src]` требуют случайного доступа только к
*вершинным* массивам, а *рёбра* можно читать строго последовательно —
значит, рёбра должны жить на диске в потоковом формате, и весь вопрос в том,
как быть с вершинными массивами.

### Вариант 1: полу-внешний (semi-external) стриминг рёбер

Вершинные массивы (rank, contrib, out-degree) целиком в RAM (~16–20 байт
на вершину), рёбра — бинарный поток на диске, читаемый последовательно
каждую итерацию.

* Плюсы: минимальная сложность, минимальный I/O (ровно 8·E байт чтения
  рёбер на итерацию), никакой лишней записи.
* Минусы: требует O(V) памяти. При V = 100 млн вершин это ~1.6 ГБ — в
  128 МБ не помещается. **Не выполняет главное условие задачи** в общем
  случае, хотя отлично работает, когда «не влезают» только рёбра.

### Вариант 2 (выбран): сеточное разбиение рёбер, edge-centric стриминг

Подход в духе GridGraph (Zhu et al., USENIX ATC'15). Диапазон вершин
делится на P интервалов по S вершин; S подбирается из бюджета так, чтобы
в памяти помещались **два** вершинных слайса. Рёбра раскладываются на диске
в сетку P×P: файл колонки j содержит рёбра с приёмником из интервала j,
сгруппированные по интервалу источника i (сегменты идут по возрастанию i).

Итерация: колонки обрабатываются по очереди; для колонки j в памяти живёт
слайс аккумулятора `next[j]`, файл колонки читается строго последовательно,
и на границе сегмента подгружается слайс вкладов `contrib[i]`. Всё I/O —
последовательное; объём памяти — жёстко O(бюджета), не зависит ни от V,
ни от E.

* Плюсы: строгий контроль памяти; только последовательный диск-I/O;
  ничего не пишется на диск в течение итерации, кроме O(V) вершинных
  массивов; при S ≥ V автоматически вырождается в вариант 1 (P = 1) —
  один код покрывает оба режима.
* Минусы: слайсы contrib читаются до P раз за итерацию — дополнительный
  I/O O(P·V) байт (см. «Ограничения»).

### Вариант 3: scatter–gather с материализацией сообщений

Подход в духе X-Stream / внешнепамятного Pregel: фаза scatter стримит рёбра
и пишет на диск поток «сообщений» (dst, вклад), сгруппированных по интервалу
приёмника; фаза gather стримит сообщения и агрегирует их в слайсы next.

* Плюсы: самый универсальный (подходит и для алгоритмов, где вклад зависит
  от данных ребра); вершинные массивы читаются ровно по одному разу.
* Минусы: каждую итерацию на диск **записывается и читается** O(E)
  сообщений (16 байт на ребро) — при power iteration с десятками итераций
  это в разы больше I/O, чем у варианта 2, плюс износ SSD записью.

**Итог:** выбран вариант 2 — он единственный одновременно (а) даёт жёсткую
гарантию по памяти при любых V и E, (б) не пишет O(E) на диск в итерациях,
(в) бесплатно включает в себя вариант 1 как частный случай.

## 3. Как устроено решение

Конвейер (один бинарник, все фазы автоматические):

```
CSV ──A──> part_*.bin ──B1──> row_i.bin ──B2──> col_j.bin + outdeg.bin + exist.bin
                                                     │
                 итерации: contrib-проход ──> обход колонок ──> rank_{0,1}.bin
                                                     │
                                              vertex,rank CSV
```

* **Фаза A — параллельный разбор CSV.** Файл делится на T байтовых
  диапазонов; поток обрабатывает все строки, *начинающиеся* в его диапазоне
  (строка начинается в p, если p = 0 или байт p−1 — `\n`) — это разбивает
  строки между потоками без пропусков и дублей при любых границах. Каждый
  поток пишет свой бинарный part-файл без какой-либо синхронизации. Разбор
  текста — самая CPU-ёмкая часть препроцессинга, поэтому распараллелена
  именно она. Парсер либерален: запятые/табуляции/пробелы, комментарии `#`,
  автоопределение заголовка — датасеты SNAP читаются напрямую.
* **Фаза B — раскладка в сетку** (две поразрядные внешние сортировки:
  по интервалу источника, затем по интервалу приёмника). Попутно, обрабатывая
  row-файл i, считаем слайс исходящих степеней (все источники строки лежат
  в одном интервале — слайс гарантированно влезает в память) и битовую карту
  существования вершин. Разбиение — чистый memcpy-конвейер, ограниченный
  диском, а не CPU, поэтому оно сознательно однопоточное.
* **Итерации** — см. вариант 2 выше. Внутри колонки рёбра обрабатываются
  параллельно: потоки пула атомарно разбирают куски буфера, аккумулируя
  `next[dst] += contrib[src]` атомарным сложением double (CAS). Чтение
  следующего 8-МБ буфера рёбер перекрывается с обработкой текущего
  (двойная буферизация). Поэлементные проходы (contrib, инициализация,
  финализация, выгрузка) работают порциями по 2 МБ — их память не зависит
  от V.
* **«Дыры» в нумерации.** Пространство id — [0, maxId]; вершины, не
  встречающиеся ни в одном ребре, отслеживаются битовой картой: они не
  получают массу, не учитываются в N и не попадают в вывод.

### Память

Бюджет `--memory-mb` — жёсткая граница для всех крупных структур программы.
Размер слайса S подбирается так, чтобы пиковая фаза (обход колонки:
next-слайс 8S + contrib-слайс 8S + два буфера рёбер) помещалась в бюджет
за вычетом фиксированного резерва 32 МБ на всё остальное (стеки потоков,
буферы поэлементных проходов, индекс сегментов, накладные расходы
аллокатора). После фаз препроцессинга освобождённая память возвращается ОС
(`malloc_trim`), и фактический пиковый RSS печатается в конце — на тестовом
прогоне 4 млн вершин / 20 млн рёбер при `--memory-mb 64` пиковый RSS
составил 45.5 МБ.

Кэш страниц ОС программой не «арендуется»: это вытесняемая память ядра,
которая по определению не входит в бюджет процесса.

### Гипер-узлы

Ключевое свойство edge-centric подхода: работа распределяется **по рёбрам,
а не по вершинам**, поэтому вершина со степенью 50 000 при средней 50 —
это просто 50 000 рёбер где-то в потоке, а не гигантский список смежности,
который нужно целиком держать в памяти или отдавать одному потоку:

* список смежности гипер-узла никогда не материализуется — рёбра размазаны
  по сегментам сетки;
* динамическая раздача кусков рёбер потокам (атомарный счётчик) даёт
  автоматический баланс нагрузки при любом перекосе степеней;
* слайс степеней при подсчёте занимает 4S байт независимо от перекоса.

Единственный эффект перекоса — конкуренция CAS-сложений на горячей ячейке
`next[v]` при экстремальной *входящей* степени; на практике она размывается
тем, что остальные приёмники распределены по слайсу. Тест генератора
(`tools/gen_graph.py --skew 3.0`) создаёт степенное распределение с
гипер-узлами и используется в тестах и бенчмарке.

### Параллелизм

* разбор CSV — по байтовым диапазонам файла, без разделяемого состояния;
* обработка рёбер в итерациях — пул потоков + динамическая раздача кусков,
  атомарная аккумуляция;
* инициализация next-слайса — параллельно по диапазонам;
* чтение с диска перекрыто с вычислениями (двойная буферизация).

Пул потоков создаётся один раз (создание потока на задачу было бы заметным
накладным расходом в горячем цикле).

## 4. Корректность и доверие к результатам

1. **Инвариант массы.** Матрица блуждания стохастическая, поэтому
   Σ rank + s(g) обязана оставаться равной 1. Значение печатается каждую
   итерацию; отклонение от 1 — верхняя оценка накопленной ошибки округления
   (на тестах ~1e-12).
2. **Сверка с независимым эталоном.** `tools/reference_leaderrank.py` —
   отдельная реализация «в лоб» на Python (другой язык, другие структуры
   данных). Тесты сверяют результаты на примере из условия и на случайном
   графе (расхождение < 1e-10).
3. **Аналитическая проверка.** Для примера из условия стационарное
   распределение решается вручную: final = (25/84, 23/84, 22/84, 14/84) ≈
   (0.297619, 0.273810, 0.261905, 0.166667) — программа выдаёт ровно это.
4. **Инвариантность к разбиению.** Результат не зависит от числа слайсов:
   тест сравнивает P = 1 и P = 6 (ручка `--slice-size`), расхождение —
   на уровне последнего бита double.
5. **Детерминизм.** Вся программа детерминирована с точностью до порядка
   суммирования чисел с плавающей точкой между потоками (различия ~1e-15,
   на ранжирование не влияют). При `--threads 1` порядок фиксирован и
   результат воспроизводится бит-в-бит; тест сверяет 1 поток против 4.
6. Все проверки собраны в `tests/run_tests.sh`.

## 5. Сложность и оценка ресурсов

Обозначения: V — размер пространства id, E — рёбра, P — число интервалов,
k — число итераций до сходимости, T — потоки, B — бюджет памяти.

* **Память:** O(B) всегда; P ≈ V·16 байт / B.
* **Препроцессинг:** время O(E/T) на разбор + O(E) на раскладку;
  диск-I/O ≈ 8E (part) + 16E (row: запись+чтение) + 16E (col) + O(V);
  всё I/O последовательное.
* **Одна итерация:** вычисления O(E/T + V); диск-I/O последовательное:
  8E (рёбра) + ≤ 8V·P (слайсы contrib, худший случай — каждая колонка
  содержит сегменты всех строк) + ~28V (rank чтение/запись, contrib
  запись, outdeg).
* **Всего:** O(k·(E + V·P)) I/O. Сходимость power iteration геометрическая
  со знаменателем |λ₂| (второе собственное число матрицы блуждания):
  k = O(log(1/ε) / log(1/|λ₂|)). Худший случай — |λ₂| → 1 (почти
  несвязные/периодические структуры; ground-вершина гарантирует
  неприводимость и апериодичность, так что сходимость всегда есть).
  На практике: 23 итерации до L1 < 1e-9 на случайном графе, 63 — на
  4-вершинном цикле из условия.
* **Точность:** double; посегментное суммирование удерживает ошибку
  контрольных сумм на уровне ~1e-12 при миллиардах слагаемых; ошибка
  на вершину пренебрежима по сравнению с tol.
* **Диск:** пик ≈ 2×8E временных файлов + 8E сетка + ~28V вершинных
  массивов; для «1 ГБ графа» из условия — единицы гигабайт на SSD.

Замер на демонстрационном прогоне (виртуальная машина, 1 vCPU, SSD;
абсолютные цифры на нормальном железе будут в разы лучше, важны
соотношения): граф V = 4·10⁶, E = 2·10⁷ (CSV 278 МБ), `--memory-mb 64`
(вершинные массивы целиком в такой бюджет не помещаются, P = 4):
разбор CSV 1.7 с, раскладка 2.5 с, ~1.3 с/итерация, всего 32 с до
сходимости, пиковый RSS 45.5 МБ. Результат бит-в-бит совпадает с прогоном
при `--memory-mb 128` (P = 1).

## 6. Ограничения подхода (и что делать дальше)

* **Слагаемое O(P·V) в I/O итерации.** При очень большом V и очень малом
  бюджете (P в сотни) чтение слайсов contrib начинает доминировать над
  чтением рёбер. Лекарства известны: contrib в float32 (минус половина
  этого I/O), двухуровневое разбиение, обход блоков по кривой Гильберта.
  Для соотношений из условия (граф ~10× бюджета) P мал и слагаемое
  несущественно.
* **Плотность нумерации.** Память масштабируется от maxId, а не от числа
  вершин: сильно разреженная нумерация (например, вершины 1 и 2³¹−1)
  раздувает V. Решение — внешняя перенумерация id (ещё одна поразрядная
  сортировка) как отдельный препроцессинг; для типичных датасетов (SNAP,
  LDBC) нумерация плотная и это не нужно.
* **Кратные рёбра не схлопываются** (интерпретируются как мультиграф —
  вклад пропорционален кратности). При необходимости дедупликация делается
  внешней сортировкой на этапе препроцессинга.
* **Атомарная аккумуляция** делает порядок суммирования зависящим от
  планировщика (см. «Детерминизм» выше). Полностью детерминированная
  многопоточность потребовала бы сортировки сегментов по приёмнику и
  раздачи потокам непересекающихся диапазонов dst.
* **Препроцессинг однопоточный** после разбора CSV — упирается в диск;
  при желании row-файлы в фазе B2 обрабатываются независимо и
  параллелятся.
* id вершин — неотрицательные int32 (по условию задачи).
